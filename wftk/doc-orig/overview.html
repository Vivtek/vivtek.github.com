The sections of a wftk system

A typical wftk application is described as a repository, a set of data sources and a description of their relationships. There are two major segments of code, which deal with the world in fundamentally different ways: the repository manager, or repmgr, sees data sources and moves data, while the wftk core sees activities in the form of processes and tasks. Both segments rely on an assortment of adaptors which interact with the external world.

Application code interacts with wftk using either the repmgr API or the wftk core API; however, since the repository manager itself knows how to invoke the wftk API in response to various events, a workflow-oriented system may have no need at all to invoke the wftk core API directly. However, for applications which want to do pure workflow without the baggage of repository management, the wftk core is carefully independent of the repmgr (many of the stock adaptors require the repmgr, but if you're not using those adaptors, you can be repository-free with no further ado.)

The repository manager also has a great deal of logic enabling it to format text and build Websites; this code can also be used to maintain other "unstructured" text using the structured data sources making up a repository. There is also code (related to the Wiki Wiki Web paradigm) which allows unstructured text to be processed into a formatted layout, enabling content producers to ignore formatting completely in many cases while writing, but still preserving their ability to affect formatting explicitly when they need to.
